# 从 0 到 1 实现 iOS UI 自动化闭环实施文档

- 版本：v1.1
- 更新时间：2026-03-20
- 适用范围：Flutter（iOS）/ Swift（iOS 原生）
- 目标读者：QA / iOS / Flutter / 后端 / 测试平台

---

## 1. 目标

这份文档要解决的，不是“怎么写几条自动化脚本”，而是“怎么把 iOS UI 自动化从 0 到 1 做成一套可复用闭环”。

这里说的闭环，至少包含：

- 自动执行核心业务流程
- 自动记录步骤与结果
- 自动保留错误证据
- 自动输出测试报告
- 为后续接口日志、异常响应、AI 分析留出扩展位

最终目标是把测试流程从“人工点击 + 人工排查”演进为：

1. 输入：测试场景、回归范围或需求描述
2. 过程：自动执行 UI 流程并采集证据
3. 输出：结构化结果、错误日志、截图、报告

---

## 2. 为什么不应该一开始就纠结“哪套框架最先进”

在 0 到 1 阶段，真正重要的不是追求“最先进框架”，而是先让下面这件事成立：

- 至少有一条执行链路能稳定跑通
- 至少有一套统一证据目录
- 至少有一套可重复执行的命令入口

如果主链路还没稳定，先去上重平台、设备池、在线报告系统，基本都会导致复杂度先行，收益滞后。

---

## 3. 先确定主执行通道

### 3.1 Swift 项目

Swift iOS 项目建议优先使用：

- `XCTest`
- `XCUITest`
- `xcodebuild`

原因很直接：

- iOS 原生一致性最强
- 与 Xcode 集成最深
- 真机、系统弹窗、原生控件行为更稳定

### 3.2 Flutter 项目

Flutter iOS 项目更实用的做法不是一刀切，而是按阶段选：

1. 如果当前目标是低侵入、快速起步、尽量不改 `lib/` 业务源码
- 优先 `flutter_driver + flutter drive`

2. 如果当前目标是贴近 iOS 原生行为、需要强系统级交互、需要和原生侧完全对齐
- 再评估 `XCUITest`

### 3.3 当前工作区已经验证过的结论

基于当前项目的实际演进经验：

- `XCUITest` 已经验证过：iOS 原生执行链路、step 截图、日志、报告闭环都能成立
- `flutter_driver` 已经验证过：在不改 `flutter_client_demo/lib/` 的前提下，也能跑通完整闭环

所以更务实的结论不是“谁绝对更好”，而是：

- Swift 原生项目：优先 `XCUITest`
- Flutter 项目：前期快速闭环优先 `flutter_driver`
- Flutter 项目：后续如要更强系统级交互，再评估 `XCUITest`

### 3.4 0 到 1 阶段不建议一开始做的事

- 不建议一开始就上重型平台
- 不建议一开始就做全量用例管理系统
- 不建议先追求多机并发
- 不建议在页面完全不可测时硬铺大量脚本
- 不建议在主链路未稳前，把精力放在炫酷报告样式上

---

## 4. 从 0 到 1 的总体实施顺序

建议严格按下面顺序推进：

1. 环境准备
2. 框架路线确定
3. 执行链路跑通
4. 页面可测性或 locator 策略确定
5. 第一条核心用例跑通
6. step、截图、日志结构化
7. 报告输出
8. 稳定性增强
9. 真实项目迁移

这几个阶段最好不要颠倒。

---

## 5. 第一阶段：环境准备

### 5.1 必备环境

最低要求：

- 完整版 `Xcode`
- `Flutter SDK`
- `CocoaPods`
- iOS Simulator 或真机

### 5.2 最低验收标准

以下命令至少应可用：

```bash
xcodebuild -version
flutter --version
pod --version
```

如果是 Flutter 项目，还应至少通过：

```bash
flutter analyze
flutter test
```

如果是 iOS 原生或 `XCUITest` 路线，还应至少能执行：

```bash
xcodebuild test ...
```

### 5.3 第一阶段产出

这一阶段结束后，应有这些结果：

- 机器环境可用
- iOS 工程能被 Xcode 正常打开
- Flutter 项目依赖正常
- 模拟器或真机能识别

---

## 6. 第二阶段：先跑通执行链路

### 6.1 先不要直接冲真实大项目

建议先做小规模 POC：

- 登录页
- 首页/发现页
- 个人页
- 1 到 2 条最核心流程

目的不是模拟真实业务复杂度，而是先验证：

- 执行器是否能稳定拉起 app
- 是否能完成输入、点击、切页
- 命令行是否可跑
- 本地与 IDE 的执行结果是否一致

### 6.2 这一阶段的核心成果

要做到：

- 有一个明确的 UI Test 入口
- 本地命令能跑通
- 至少一条流程完整通过

如果这一步没跑通，后面所有“日志、报告、AI 分析、平台化”都不成立。

---

## 7. 第三阶段：页面可测性治理或 locator 策略治理

### 7.1 为什么这是核心前置条件

UI 自动化最本质的问题是：

机器如何稳定找到要操作的控件。

如果页面没有可测性，或者 locator 策略不统一，自动化仍然能写，但会非常脆弱：

- 文案一改就挂
- 布局一改就挂
- 列表顺序一变就挂
- Flutter 页面树或 iOS Accessibility 节点重建后定位失效

### 7.2 如果走 XCUITest

重点解决：

- `accessibilityIdentifier`
- 页面级稳定 selector
- 页面 ready 条件
- 元素可点击性判断

这类治理通常会涉及客户端少量可测性改造。

### 7.3 如果走 Flutter 自测路线

对 iOS Accessibility 树依赖会降低，但不代表完全不用治理。

重点要保证：

- 页面文案不要过度易变
- locator 统一收口到 `pages/`
- case 层不直接堆满定位细节
- 等待、重试、失败处理统一收口到 `support/`

### 7.4 对开发侧的要求

建议推动形成规范：

- 关键交互控件必须可被稳定识别
- 关键状态展示区必须可判断
- 列表 item、弹窗、浮层、导航不要完全依赖视觉结构

### 7.5 如果真实项目暂时不具备可测性怎么办

不是不能做，而是应该这样推进：

1. 先做可测性评估
2. 先选 1 到 2 条主流程硬跑
3. 产出“控件可测性缺口清单”
4. 推动开发按优先级补齐

不要在完全不可测的页面上一次性铺大量脚本。

---

## 8. 第四阶段：实现第一条核心 UI 自动化

### 8.1 用例范围建议

第一条用例只做最核心冒烟链路，例如：

- 启动 app
- 登录
- 进入首页
- 搜索或浏览
- 进入个人页
- 退出登录

### 8.2 代码组织建议

如果是 `XCUITest`：

- `RunnerUITests.swift`
- `cases/`
- `pages/`
- `support/`

如果是 `flutter_driver`：

- `test_driver/app.dart`
- `test_driver/main.dart`
- `cases/`
- `pages/`
- `support/`

### 8.3 为什么要 page object

页面对象的价值在于：

- locator 统一维护
- 页面切换逻辑统一收口
- 控件不稳定时只改一处
- 后续迁移真实项目更容易按模块维护

---

## 9. 第五阶段：结构化步骤、日志与截图

这一步是从“能跑”进入“可排查”的关键。

### 9.1 step 化

每条流程都建议拆成：

- `S001`
- `S002`
- `S003`

每个 step 都要能回答：

- 当前做了什么
- 成功还是失败
- 失败发生在哪一步

### 9.2 运行上下文

建议每次执行都生成：

- `run_id`
- `step_id`
- 时间戳

后续如果接网络采集，再增加：

- `req_id`

### 9.3 截图策略

建议至少做两层：

1. 常规截图
- 每个 step 后截图

2. 异常截图
- 点击失败截图
- 断言失败截图
- 页面异常截图
- 接口异常截图

### 9.4 最低落地标准

即使暂时没有接口日志，也至少要做到：

- 知道每一步执行了什么
- 知道失败发生在哪
- 能看到当时页面截图

---

## 10. 第六阶段：接口日志与异常响应采集

### 10.1 为什么要做

只看 UI 结果，不足以定位问题。

真正有价值的闭环一定要把下面这些东西串起来：

- UI 操作
- 接口请求
- 接口响应
- 错误信息

### 10.2 采集建议

建议在客户端网络层统一记录，至少包含：

- `ts_ms`
- `run_id`
- `step_id`
- `req_id`
- `url`
- `method`
- `status`
- `biz_code`
- `resp_ms`
- `params`
- `resp`
- `err`

### 10.3 与 UI 步骤的关联

关键不是“记录接口”，而是“知道哪个 step 触发了哪个请求”。

建议：

- Runner 每进入一个 step，更新当前 `step_id`
- 网络层读取当前 `step_id`
- 每个请求生成自己的 `req_id`

最终形成：

`run_id -> step_id -> req_id -> request/response -> screenshot`

### 10.4 这一阶段的边界

如果当前只是 POC，没有真实后端请求，也可以先做：

- 日志结构设计
- 占位文件格式
- 报告消费方式

真实网络采集可以在迁移真实项目时再落地。

---

## 11. 第七阶段：报告生成

### 11.1 报告目标

报告不只是“通过 / 失败”，还要能回答：

- 用例执行了哪些步骤
- 失败发生在哪一步
- 对应截图是什么
- 是否有接口异常
- 是否有关键字段缺失
- 建议排查方向是什么

### 11.2 推荐分阶段实现

第一阶段先做轻量报告：

- Markdown 报告
- HTML 报告
- 单文件 HTML 报告

第二阶段再考虑增强：

- AI 分析
- 图文混合报告
- 规则预检 + 智能总结

### 11.3 为什么不建议一开始就上重工具

像 Airtest 或其他在线报告工具，可以评估，但不要一开始就作为主线：

- 会增加体系复杂度
- 容易把注意力从“稳定执行链路”转移到“平台工具”
- 如果主链路还没稳，报告做得再漂亮也没意义

建议先用本地可控的 Markdown / HTML 跑通，再决定是否引入更重的工具。

## 12. 推荐目录与职责划分

### 12.1 当前推荐目录

```text
repo-root/
  qa/
    Makefile
    configs/
    scripts/
    analyzer/
    reports/
  flutter_client_demo/
    lib/
    test/
    test_driver/
      app.dart
      main.dart
      cases/
      pages/
      support/
    ios/
```

### 12.2 当前职责划分

`qa/`

- 统一命令入口
- 统一配置入口
- 统一报告生成
- 统一运行结果归档

`test_driver/` 或 `RunnerUITests/`

- 测试入口
- 用例分发
- 页面对象
- 公共 runner、等待、重试、失败处理

`qa/reports/<run_id>/`

- 本地证据仓库
- 保存日志、截图、结果 JSON、Markdown/HTML 报告

### 12.3 团队分工建议

QA / 自动化：

- 编写用例
- 维护 page object
- 维护执行脚本
- 维护报告模板

iOS / Flutter 开发：

- 补关键控件可测性
- 预留网络层采集点
- 对不稳定控件结构做治理

后端 / 平台：

- 提供测试环境支持
- 提供异常数据、错误码说明
- 后续配合日志关联与报告归因

- 形成稳定 locator / selector 策略

### 阶段 3：2 到 4 周

目标：

- 引入 step 级日志

目标：

## 14. 当前项目已经验证的关键经验

基于当前工作区里的 demo，已经验证过这些结论：

1. `XCUITest` 可以承载 iOS 原生执行、step 截图、日志和报告闭环
2. `flutter_driver` 可以在不改 `flutter_client_demo/lib/` 的前提下跑通 Flutter UI 自动化闭环
3. `qa/` 层作为统一执行中台是成立的，最值得复用
4. 真正可迁移的不是 demo 页面本身，而是：
   - 目录结构
   - 命令入口
   - `run_id`
   - 日志格式
   - 截图格式
   - 报告链路

## 15. 风险与应对

### 15.1 最大风险：控件不可测或 locator 策略不统一

风险：

- 没有稳定标识
- 页面轻微改动就导致脚本挂掉

应对：

- 关键控件优先补可测性
- locator 统一收敛到 `pages/`
- 页面结构变化时优先只改页面对象层

### 15.2 第二风险：只做执行，不做留证

风险：

- 脚本失败了但无法排查

应对：

- 必须有 step
- 必须有截图
- 必须有失败日志
- 必须有结果 JSON

### 15.3 第三风险：过早平台化

风险：

- 工程复杂度升高
- 主链路未稳，投入产出比低

应对：

- 先做本地闭环
- 再做平台化、设备池、多机并发

---

## 16. 一句话结论

从 0 到 1 落地 iOS UI 自动化，正确路线不是先堆工具，而是：

先选定主执行链路，再治理页面可测性或 locator 策略，再把步骤、截图、日志、报告逐步补齐，最终形成可扩展到真实大项目的闭环体系。

这套路线的关键成功条件只有两个：

- 执行链路稳定
- 关键控件或关键定位能力稳定

只要这两点成立，后面的日志、接口采集、报告、AI 分析都可以持续演进。
