# UI 自动化稳定性增强清单

- 版本：v1.1
- 适用范围：iOS `XCUITest` / Flutter `flutter_driver`
- 目标：把“能跑”的自动化提升为“稳定可持续运行”的自动化

---

## 1. 背景

UI 自动化从 0 到 1 跑通后，通常还只是“可用雏形”，离“稳定工程化”还有明显距离。

最常见的问题不是脚本不会写，而是：

- 页面还没准备好就点击
- 元素存在但不可交互
- 动画未结束
- 页面局部刷新导致元素句柄失效
- 网络慢一点就断言超时
- Flutter 页面树或 iOS Accessibility 节点重建后，旧 locator 失效

因此，稳定性增强的重点不是继续堆更多用例，而是补齐下面这些“机制层”：

- 等待机制
- 判断机制
- 重试机制
- 超时机制
- 失败留证机制
- 结构化日志机制
- locator 稳定性策略

---

## 2. 稳定性建设目标

一套可长期维护的 UI 自动化，至少要做到：

1. 不因为偶发加载波动频繁误失败
2. 不因为单次点击失误直接整条流程报废
3. 失败时能快速定位到具体 step、动作和页面
4. 同一套机制可以扩展到更多页面和更复杂业务链路
5. 同一条主流程重复运行时，通过率保持稳定

---

## 3. 等待机制

等待机制是稳定性的第一层。

### 3.1 必须区分的 3 种状态

不能只判断“元素存在”，还要区分：

- 元素存在
- 元素可交互
- 页面真正就绪

这三种状态在 `XCUITest` 和 `flutter_driver` 里都不能混为一谈。

### 3.2 推荐等待层级

建议按以下优先顺序设计等待：

1. 页面级等待
- 页面标题
- 页面主按钮
- 页面核心输入框
- 页面核心列表或结果区

2. 动作前等待
- 点击前等待可交互
- 输入前等待可聚焦
- 滑动前等待容器出现

3. 动作后等待
- 点击登录后等待首页 ready
- 点击筛选后等待结果区刷新完成
- 点击保存后等待成功提示或状态变化

### 3.3 推荐实现方式

建议每个页面对象都定义一个“页面 ready 条件”，例如：

- `LoginPage.waitForVisible()`
- `DiscoverPage.waitForCatalog()`
- `ProfilePage.waitForVisible()`

所有 case 都应该基于页面 ready 条件往下执行，而不是直接往下点。

### 3.4 不推荐做法

不推荐长期依赖：

- 固定 `sleep`
- 所有场景统一等待同一个超时时间
- 只判断存在，不判断页面是否真正切换成功

固定 `sleep` 只能作为临时兜底，不能作为主策略。

---

## 4. 判断机制

判断机制决定的是：什么时候可以继续往下走。

### 4.1 点击前判断

点击前至少判断：

- 元素存在
- 元素可交互
- 当前页面没有明显遮挡层
- 当前不在 loading、toast 或动画阻塞状态

### 4.2 点击后判断

点击后要判断“动作是否真的成功”，而不是只认为“点击没报错”就算成功。

例如要判断：

- 页面是否切换
- 结果文案是否变化
- 按钮状态是否变化
- 列表内容是否刷新
- 数量或状态是否变化

### 4.3 输入后判断

输入后建议判断：

- 当前输入框值是否更新
- 输入是否触发了预期结果
- 键盘或输入焦点是否已退出阻塞状态

### 4.4 页面切换判断

不要只依赖“按钮点下去了”来判断切页成功。

正确做法是：

- 使用目标页面的 ready 条件判断是否到达目标页面

---

## 5. 重试机制

重试机制是稳定性的第二层，但必须有边界。

### 5.1 哪些动作适合重试

适合重试：

- 点击
- 输入
- 页面 ready 等待
- 局部结果刷新等待

不建议盲目重试：

- 会触发重复提交的操作
- 会产生副作用的关键请求
- 已经明显进入错误状态的流程

### 5.2 推荐重试策略

建议采用：

- 最大重试次数：2 到 3 次
- 每次重试前重新查找元素
- 重试前重新判断页面状态
- 重试间隔使用短等待

### 5.3 为什么要“重新获取元素”

在 Flutter 和 iOS 场景里，经常会出现：

- 页面刷新
- 节点重建
- 旧元素句柄失效

所以重试不能直接复用旧元素对象，应该重新查询。

### 5.4 推荐兜底方式

点击失败后的兜底顺序建议：

1. 重新查找元素
2. 再判断存在与可交互
3. 再尝试点击
4. 仍失败则截图并中断当前 step

---

## 6. 超时机制

超时策略不能一刀切。

### 6.1 建议分层超时

建议至少区分：

- 页面级超时：8 到 15 秒
- 动作级超时：3 到 5 秒
- 结果刷新级超时：5 到 10 秒

### 6.2 超时设计建议

- 登录页加载超时，不应和按钮点击超时相同
- 页面首屏加载可更长
- 元素点击等待可更短
- 搜索结果刷新可用中间值

### 6.3 超时后必须做的事

超时不是简单报错结束，至少应：

- 截图
- 记录当前 `step_id`
- 记录动作名
- 记录错误信息
- 记录页面调试信息

---

## 7. 失败处理机制

失败处理是“可排查”的关键。

### 7.1 每次失败都应留证

至少留这些证据：

- 当前截图
- 当前 `step_id`
- 当前动作名
- 当前页面描述
- 错误信息
- 时间戳

### 7.2 推荐失败时采集的内容

建议失败时统一采集：

1. 页面截图
2. 调试描述
3. 最近一步动作日志
4. 当前运行参数
5. 当前测试名

### 7.3 中断策略

建议默认：

- 关键流程失败后立即中断
- 不要在明显失败状态下继续跑后续步骤

否则后续错误大概率都是连锁噪音。

---

## 8. 日志机制

日志是把“执行过程”结构化的关键。

### 8.1 最低要求

每个 step 至少记录：

- `run_id`
- `step_id`
- step 名称
- 动作类型
- 开始时间
- 结束时间
- 结果
- 失败原因

### 8.2 推荐动作日志

建议至少记录这些动作：

- click / tap
- input
- swipe
- wait
- assert
- navigate

### 8.3 后续与接口日志关联

如果后续接入网络层日志，建议继续扩展：

- `req_id`
- request / response 摘要
- `biz_code / status`
- 错误码

最终形成：

`run_id -> step_id -> action log -> req_id -> screenshot`

### 8.4 当前目录落位建议

如果按当前项目结构实现，推荐：

- `cases/`：只写业务流程
- `pages/`：放 locator 与页面级操作
- `support/`：放日志写入、失败留证、等待和重试
- `qa/reports/<run_id>/logs/`：放 step / action / result 文件

---

## 9. Selector / Locator 稳定性策略

稳定 locator 是长期维护的基础。

### 9.1 优先级建议

优先使用：

1. 稳定可测 id
2. 明确且不易变化的主文案
3. 结构化区域
4. 最后才是索引定位

### 9.2 不推荐依赖

不建议长期依赖：

- 第几个按钮
- 第几个输入框
- 纯坐标点击
- 容易变化的展示文案

### 9.3 Flutter 特别注意

Flutter 场景下要特别警惕：

- widget 结构变化
- 输入框与标签分离
- 多个相同文本共存
- 列表或过滤器刷新后节点重建

所以 locator 设计必须务实：

- 在 `pages/` 中统一定义
- 尽量做页面级封装
- 允许必要的兜底 locator，但不要到处散落

### 9.4 XCUITest 特别注意

XCUITest 场景下要特别警惕：

- 只存在 accessibility 代理节点
- 元素 `exists == true` 但不可点击
- 页面层级变化后旧 selector 失效

因此：

- 页面 ready 条件必须存在
- 点击前必须判断可交互
- 失败时要保留 `debugDescription`

---

## 10. 页面对象建议

建议每个页面对象都具备这些能力：

- 页面 ready 判断
- 核心控件访问
- 页面级动作方法
- 页面级断言方法

### 10.1 不建议

不建议把所有 locator 和动作都堆到测试方法里。

否则会导致：

- locator 分散
- 重试逻辑重复
- 等待逻辑重复
- 改一个页面要改很多地方

### 10.2 当前目录推荐

如果是 Flutter 自测框架：

- `test_driver/pages/`

如果是 XCUITest：

- `RunnerUITests/pages/`

都应该只负责页面对象本身，不负责报告渲染或命令编排。

---

## 11. 推荐公共能力抽象

建议逐步抽出这些公共层：

1. `waits`
- 等待存在
- 等待可交互
- 等待页面 ready

2. `retry`
- 带重试点击
- 带重试输入
- 带重试页面切换

3. `assertions`
- 页面断言
- 状态断言
- 列表断言

4. `failure_handler`
- 失败截图
- 调试描述
- 失败上下文写入

5. `artifact_writer`
- step 日志
- action 日志
- 结果 JSON

6. `locator_strategy`
- locator 组合查找
- locator 兜底策略

### 11.1 当前建议的落位

如果按当前项目目录继续扩展，建议：

```text
test_driver/support/
  case_runner.dart
  driver_factory.dart
  waits.dart
  retry.dart
  assertions.dart
  failure_handler.dart
  locator_strategy.dart
  network_capture.dart
```

如果是 XCUITest，则对应放到：

```text
RunnerUITests/support/
  waits.swift
  retry.swift
  assertions.swift
  failure_handler.swift
  artifact_writer.swift
  locator_strategy.swift
```

---

## 12. 报告层建议

稳定性增强之后，报告层至少要展示：

- 用例名称
- 执行步骤
- 每步耗时
- 每步结果
- 错误原因
- 对应截图

### 12.1 最小落地建议

先做：

- Markdown 报告
- HTML 报告
- 单文件 HTML 报告

### 12.2 目录建议

报告和证据统一写到：

- `qa/reports/<run_id>/`

好处是：

- 每次运行天然隔离
- 可以快速打包和回放
- 报告、日志、截图天然同目录关联

---

## 13. 建议的实施优先级

建议按下面顺序补稳定性：

1. 页面 ready 判断
2. 点击前后判断
3. 元素重查 + 重试
4. 超时策略分层
5. 失败截图和调试描述
6. step 级结构化日志
7. 接口日志关联
8. Markdown / HTML 报告

---

## 14. 最低验收标准

如果要说“这套 UI 自动化进入可用稳定态”，至少应满足：

1. 同一条用例连续执行多次通过率稳定
2. 页面慢一点不会立即误失败
3. 点击失败时能自动重试
4. 失败时有截图和错误日志
5. 报错后能定位到具体 step

### 14.1 建议增加的量化门槛

建议补充这些量化观察指标：

- 同一条主流程连续运行 5 次，结果稳定
- 失败时 100% 能定位到 step 和截图
- 新增用例时，不需要修改报告层和命令层
- 页面改动后，优先只改 `pages/` 层即可恢复

---

## 15. 一句话结论

UI 自动化真正难的不是写点击脚本，而是把等待、判断、重试、失败留证、稳定 locator 这些机制层补齐。

没有这些，自动化只能算演示版；  
补齐这些之后，才具备往真实大项目扩展的基础。
